## 1. 文档目的

本文档定义当前 MVP 的运行时状态机。

其重点不是表达完整叙事，而是回答：

- 程序在运行时有哪些顶级状态
- 玩家在哪些条件下从一个状态进入另一个状态
- 哪些数据在状态切换时保留，哪些必须重置
- `PhysicalMode` 与 `EchoMode` 如何共存于同一主场景
- 回溯如何在不实现全历史时间系统的前提下成立

## 2. 设计目标

### 2.1 总目标

当前运行时状态机必须支撑以下主体验闭环：

1. 进入世界
2. 在 `PhysicalMode` 探索
3. 消耗资源并推进
4. 死亡或耗尽触发回溯
5. 回溯后保留残留层
6. 经作者定义条件进入 `EchoMode`
7. 在 `EchoMode` 中对残留层执行有限干预
8. 导向至少一个结局

### 2.2 技术目标

状态机设计必须满足：

- 结构清晰
- 行为可重演
- 重置边界明确
- 不依赖复杂时间倒放
- 可与 `L2-02` 的单一主场景方案对接

## 3. 状态机设计原则

### 3.1 顶级状态少而明确

当前 MVP 不应为每个章节单独建立完全独立的顶级 runtime state。

理由：

- 会放大实现复杂度
- 会让“十章节结构”与“模式结构”纠缠
- 与单主场景复用路线冲突

因此，当前设计优先使用：

- 少量顶级状态
- 多数叙事进度放入运行时上下文
- 模式切换由规则驱动

### 3.2 模式是状态，但进度不是顶级状态

当前设计区分：

- `Top-Level State`
- `Mode`
- `Progress Phase`

其中：

- 顶级状态控制系统生命周期
- 模式控制可用行为集
- 进度阶段控制关卡解释和叙事触发

### 3.3 回溯是“状态回滚 + 残留保留”

当前设计明确拒绝：

- 全历史录像
- 全世界逐帧倒放
- 大规模可逆物理模拟

当前设计采用：

- 锚点式重置
- 局部持久化
- 作者控制的残留层

## 4. 顶级状态定义

当前推荐的顶级状态如下：

- `Boot`
- `FrontEnd`
- `LoadingLevel`
- `PhysicalMode`
- `DeathReset`
- `EchoMode`
- `Ending`
- `TerminalError`

### 4.1 `Boot`

职责：

- 初始化程序
- 初始化配置
- 初始化必要资源管理器

退出条件：

- 启动成功后进入 `FrontEnd`
- 启动失败进入 `TerminalError`

### 4.2 `FrontEnd`

职责：

- 呈现最小开始界面或启动入口
- 接收“开始体验”的动作

退出条件：

- 玩家开始游戏 -> `LoadingLevel`

### 4.3 `LoadingLevel`

职责：

- 加载关卡包
- 构建 `StaticLevel`
- 构建初始 `SessionState`
- 建立初始锚点

退出条件：

- 成功 -> `PhysicalMode`
- 加载失败 -> `TerminalError`

### 4.4 `PhysicalMode`

职责：

- 承载前半段主探索循环
- 允许真实移动、声呐使用、资源消耗、基础触发
- 响应“工具有罪”的感知偏差逻辑

典型进入来源：

- `LoadingLevel`
- `DeathReset`

典型退出条件：

- 资源耗尽或死亡 -> `DeathReset`
- 达到 Echo 切换条件 -> `EchoMode`
- 达到结束条件 -> `Ending`

### 4.5 `DeathReset`

职责：

- 冻结当前运行
- 记录必要残留
- 将会话状态回滚到最近锚点
- 重建可重建的运行时状态

进入来源：

- `PhysicalMode`

退出条件：

- 回滚成功 -> `PhysicalMode`
- 回滚失败 -> `TerminalError`

### 4.6 `EchoMode`

职责：

- 承载后半段有限干预循环
- 复用同一主场景
- 改变玩家可操作集合
- 允许对残留层进行删改、扰动、重标记或引导

进入来源：

- `PhysicalMode`

退出条件：

- 达到结局条件 -> `Ending`
- 需要返回前态的特殊情况目前不在 MVP 范围内

### 4.7 `Ending`

职责：

- 结算最终状态
- 播放最小结局表现
- 允许退出或回到前端

### 4.8 `TerminalError`

职责：

- 对严重不可恢复问题进行明确失败输出
- 避免程序继续处于不一致状态

## 5. 模式与进度分层

### 5.1 模式

当前 MVP 只有两个正式模式：

- `Physical`
- `Echo`

### 5.2 进度阶段

为了将创意中的 `54321 -> 12345` 与低成本状态机对齐，当前推荐用进度阶段表达章节结构，而不是用十个顶级状态表达。

当前建议的最小阶段模型：

- `Descend5`
- `Descend4`
- `Descend3`
- `Descend2`
- `Descend1`
- `Echo1`
- `Echo2`
- `Echo3`
- `Echo4`
- `Echo5`

这些阶段：

- 存在于 `SessionState`
- 影响当前偏差规则、触发器有效性、残留解释和可用事件
- 不等同于完整场景切换

### 5.3 当前 MVP 推荐降级实现

若十阶段在 MVP 中成本过高，允许先降级为以下抽象阶段：

- `EarlyPhysical`
- `MidPhysical`
- `LatePhysical`
- `EarlyEcho`
- `LateEcho`

但降级必须在产品上被接受，且不得破坏整体体验闭环。

## 6. 核心运行时数据

### 6.1 Runtime 数据分层

运行时数据分为三类：

#### 静态关卡数据

来自 `StaticLevel`，在单次关卡加载后不应被随意重建：

- 几何
- 碰撞
- marker
- volume
- 规则定义

#### 会话状态

描述当前玩家本轮会话的状态：

- 当前模式
- 当前进度阶段
- 当前锚点
- 当前能量
- 玩家位置与朝向
- 已触发事件集
- 已解锁模式条件

#### 持久残留状态

跨回溯保留：

- 残留痕迹
- Echo 线索状态
- 历史干预结果
- 已暴露的部分真相标签

### 6.2 SessionState 最小字段

当前建议 `SessionState` 至少包含：

- `mode`
- `phase`
- `current_anchor_id`
- `energy`
- `player_transform`
- `trigger_flags`
- `ending_flags`

### 6.3 ResidueState 最小字段

当前建议 `ResidueState` 至少包含：

- `retained_echoes`
- `retained_marks`
- `phase_memory_flags`
- `echo_intervention_flags`

## 7. 状态切换事件

### 7.1 从 `FrontEnd` 到 `LoadingLevel`

触发事件：

- `StartGame`

### 7.2 从 `LoadingLevel` 到 `PhysicalMode`

触发条件：

- `StaticLevel` 构建成功
- 初始 `SessionState` 初始化成功

### 7.3 从 `PhysicalMode` 到 `DeathReset`

触发事件可以包括：

- `EnergyDepleted`
- `FatalHazardHit`
- `ForcedResetByRule`

### 7.4 从 `DeathReset` 到 `PhysicalMode`

触发条件：

- 残留写入成功
- 锚点回滚成功
- 玩家被重置到合法状态

### 7.5 从 `PhysicalMode` 到 `EchoMode`

该切换必须由作者定义事件驱动，而不是纯系统自动推断。

当前推荐条件：

- 达到特定阶段门
- 满足特定残留条件
- 触发特定真相边界事件

### 7.6 从 `EchoMode` 到 `Ending`

触发条件应至少包含以下一类：

- 完成关键干预
- 抵达指定结局点
- 达成指定结局标志组合

## 8. 回溯设计

### 8.1 锚点模型

当前运行时采用锚点模型。

锚点由内容层和规则层共同定义：

- 空间位置来自 `StaticLevel`
- 使用条件来自规则层

### 8.2 回溯流程

`DeathReset` 的推荐流程如下：

1. 冻结当前输入与交互
2. 记录导致回溯的原因
3. 计算并写入新的残留层数据
4. 选择目标锚点
5. 重建 `SessionState`
6. 恢复到 `PhysicalMode`

### 8.3 回溯后保留项

当前推荐保留：

- 残留层
- 已解锁的阶段记忆标记
- 某些叙事触发历史

### 8.4 回溯后重置项

当前推荐重置：

- 玩家即时位置
- 玩家即时朝向
- 即时能量值
- 本轮局部触发缓存
- 本轮短期临时可见结果

### 8.5 不允许的回溯行为

当前 MVP 不允许：

- 在回溯中保留所有物理细节
- 在回溯中保留所有一次性演出状态
- 在回溯中逐帧再现历史轨迹

## 9. PhysicalMode 设计

### 9.1 行为集合

`PhysicalMode` 应允许：

- 移动
- 观察
- 声呐发射
- 基础触发交互

### 9.2 系统约束

`PhysicalMode` 中以下系统必须生效：

- 能量消耗
- 感知偏差规则
- 世界触发规则
- 死亡/耗尽检测

### 9.3 体验目标

该模式应强化：

- 对工具的依赖
- 对空间的误读
- 对残留的初步利用

## 10. EchoMode 设计

### 10.1 模式定位

`EchoMode` 不是第二套完整玩法系统，而是对前半段主场景和残留层的低成本重读模式。

### 10.2 行为集合

当前推荐 `EchoMode` 允许的最小行为包括：

- 浏览残留痕迹
- 激活特定 Echo 节点
- 删除或削弱部分错误痕迹
- 标记或放大部分引导痕迹
- 触发特定模式限定事件

### 10.3 当前不建议支持的行为

当前 MVP 不建议在 `EchoMode` 支持：

- 自由建造
- 完整对象操控
- 通用物理推拉
- 复杂战斗
- 独立地图漫游

### 10.4 与 PhysicalMode 的关系

`EchoMode` 应建立在同一场景对象 ID 和同一残留层之上。

不得为 `EchoMode` 构建完全平行、完全独立的第二运行时世界。

## 11. 失败处理

### 11.1 Fatal 状态切换错误

以下情况应视为 fatal：

- 初始关卡加载成功但缺少起始锚点
- `DeathReset` 找不到合法锚点
- `EchoMode` 进入条件触发但缺少必须数据
- 状态恢复后玩家位于非法空间
- 模式切换后关键运行时结构缺失

### 11.2 恢复策略

对 fatal 状态切换错误，当前策略为：

- 输出可定位错误
- 停止继续游戏逻辑
- 进入 `TerminalError`

## 12. 最小验证清单

实现本状态机后，至少应验证：

1. 程序可从 `Boot` 进入 `FrontEnd`
2. 可从 `FrontEnd` 进入 `LoadingLevel`
3. 可成功进入 `PhysicalMode`
4. 在能量耗尽时可进入 `DeathReset`
5. 在 `DeathReset` 后可回到合法的 `PhysicalMode`
6. 回溯后残留层仍然存在
7. 可在作者定义条件下进入 `EchoMode`
8. 可从 `EchoMode` 抵达 `Ending`
9. 任一关键状态缺失时会进入 `TerminalError`

## 13. 决策记录

### D-001

- 决策：采用少量顶级状态 + 进度阶段，而不是十个独立顶级章节状态
- 可选方案：十章节分别独立状态机
- 未选原因：实现成本高，且与单主场景复用方案冲突
- 失效条件：若后续明确改为多关卡强分离结构

### D-002

- 决策：回溯采用“锚点回滚 + 残留保留”
- 可选方案：完整时间回放或全历史记录
- 未选原因：超出 MVP 成本承受范围
- 失效条件：若后续项目范围显著扩大

### D-003

- 决策：EchoMode 作为同场景复用模式而非第二独立世界
- 可选方案：单独 Echo 地图或完全独立后半段关卡
- 未选原因：会显著放大内容与实现成本
- 失效条件：若后续明确放弃当前 MVP 范围

## 14. 待决事项

当前尚未冻结：

- 十章节是否在 MVP 中完整显式表现
- 每章死亡次数限制的具体 runtime 表达

- 结局条件的最终组合方式

---

本设计的核心原则是：

**把复杂的叙事结构压缩为可控的状态切换，把复杂的时间概念压缩为可验证的回溯语义。**
