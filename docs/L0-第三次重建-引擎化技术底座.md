## 1. 这一次为什么必须重做

当前 `SonarJam` 的架构债是真的，列出来不是骂自己，是让未来知道**哪些痛是结构性的、绕不开的**：

### 1.1 渲染管线是拼凑

- `miniquad` GL_POINTS 自己开 default_pass + 手写 shader
- depth pre-pass 写在外面用 macroquad 的 draw_mesh
- ship::render 是另一套独立路径
- 三套渲染同时活着，互相耦合通过 `set_camera` / `set_default_camera` 切上下文
- 想加 PBR、normal map、shadow map、HDR、bloom — 每一项都得动这个脆弱的拼接

### 1.2 没有 ECS / scene graph

- `GameApp` 一个 struct 现在 30+ 字段，覆盖 mode、世界、玩家、声呐、转场、字幕、BT、tutorials、低电定时器……
- 状态之间没有隔离，借用冲突频繁（每次加新功能至少改 3 处 `&self.worlds[self.current_idx]` 的写法）
- 实体动态生成（BT、leak tree 圆柱、未来的捡到物品）没有统一容器

### 1.3 内容管线靠命名 + 脚本，不可视化

- Blender 里靠 `R_/C_/P_/H_/M_/V_` 前缀约定身份
- 改 marker 位置要重跑 Blender 脚本、重导 GLB、重 commit 70MB 文件
- 没有可视化预览，错放一个 marker 在虚空里 = 进游戏才发现
- 关卡数据散在 GLB 里，元数据（哪段剧本触发、哪个 trigger volume）无处安放

### 1.4 剧本/字幕硬编码

- 字幕全在 `src/narrative/mod.rs` 的 match 里
- 改一句台词要 cargo build（10s+）
- 时间编排（delay / hold）也是代码字面量
- 剧本作者必须懂 Rust
- 调节奏靠"改一句、编译、跑、再改" 反馈延时高得离谱

### 1.5 关卡是常量数组

```rust
const PHASE_WORLD_PATHS: [&str; 5] = [ ... ];
fn world_index_for_phase(phase: u32) -> usize { ... }
```

加一个新关卡 = 改源码、改函数、改 match arm。剧情每加一段都是侵入式修改。

### 1.6 移植性差

- `mouse_left_down` / `mouse_right_down` 直接 link `user32.dll`
- spawn 用 `SetCursorPos` 钉光标也是 win32
- 音频用 `mem::forget(stream)` 绕 Windows drop panic
- 这些不动，第三次跨平台死翘翘

### 1.7 资产无热重载

每改一张图、一段音、一个 GLB → 完整重启。
导致迭代节奏被打断 → 调参靠记忆而不是直觉。

## 2. 第三次要立的两条铁律

任何技术选择，凡是违反这两条，就是错的选择。

### **铁律 A：任何反馈在 5 秒内可见**

- 改字幕 → 文件保存 → 游戏内立刻更新（不重启、不重编）
- 改光照参数 → 滑滑条 → 即时
- 改剧本时间 → 改数字 → 当前段落立刻按新时间走
- 改 marker 位置 → 在编辑器拖动 → 当前帧立刻反映

### **铁律 B：非程序员能编辑大部分东西**

- PA 自己（设计者）能在编辑器里：
  - 放 marker、移动 marker、删 marker
  - 写剧本、改剧本时间、改卡片样式
  - 调音效触发条件
  - 改关卡里"哪个 marker 触发哪段台词"
- 美术（如果有）能：
  - 替换 GLB 资产、立即看到
  - 改贴图、改光照
- 程序员只在：
  - 加新组件类型 / 新玩法系统
  - 加新引擎能力（光照算法、新 shader）
  - 修引擎 bug

## 3. 推荐技术栈（带选择理由）

不是"行业最佳实践"，是"最适合 SonarJam 第三次"的选择。

| 层 | 选 | 为什么 | 备选 |
|---|---|---|---|
| **窗口/输入** | `winit` 0.30+ | 生产级、跨平台、和 wgpu 配对成熟 | sdl2（不爱 C 依赖） |
| **GPU API** | `wgpu` 22+ | Rust 生态最稳的 GPU 抽象，跨 Vulkan/Metal/DX12/GL | raw OpenGL（要死） |
| **ECS** | `bevy_ecs` 单 crate | API 成熟、并行调度、不绑 bevy 全家桶 | hecs（更轻但 query API 没 bevy 香） |
| **shader** | WGSL（wgpu 原生）+ naga | 不要 GLSL→SPIR-V→Naga 转 N 次 | HLSL（要工具链） |
| **物理** | `rapier3d` | Rust 主力物理库，3D 凸包/triangle mesh/character controller 都有 | parry+ 自己手撸（轮子） |
| **音频** | `kira` 0.10+ | 比 rodio 现代，明确的 sound handle、tween、空间化 | rodio（就是现在用的，但 hack 太多） |
| **UI（编辑器）** | `egui` 0.30 + `egui-wgpu` | 立即模式 UI、和 wgpu 集成成熟 | iced（陡） |
| **场景文件** | RON（Rusty Object Notation） | 跟 Rust struct 一一对应，serde derive 即用 | JSON（无 enum 友好）/ YAML（缩进易错） |
| **资产** | 用 image / gltf / hound 直接读 + 自己包一层 AssetServer + notify watcher | 不引入巨石 bevy 全家桶 | bevy_asset（绑 bevy） |
| **剧本 DSL** | 自定义文本格式（参见 §4.4），用 nom 或手写解析器 | 比 lua 嵌入轻 | luau / rhai / mlua（功能溢出） |
| **日志** | `tracing` + `tracing-subscriber` | 结构化日志、span 化 | env_logger（够用但弱） |

### 不要做的事

- ❌ 不要用 bevy 全家桶 —— 大、风格强、不可剥离
- ❌ 不要自己写 ECS —— bevy_ecs 已经够好
- ❌ 不要写 lua/wasm 脚本绑定 —— 自定义 DSL 解析 100 行能搞定，绑虚拟机 1000+ 行
- ❌ 不要用 egui 做游戏内 HUD —— 留给编辑器，游戏内 HUD 自己渲染（已有 macroquad 思路可参考）

## 4. 关键架构决策

按"用户问感的从外到内"展开。

### 4.1 模式：双进程合一

一个 binary，两个模式：
- **游戏模式**（默认）：玩家视角，全部 HUD + 真实玩法
- **编辑器模式**（F11 / `--editor` 启动）：第三人称漫游 + egui 面板 + scene 直接保存

编辑器和游戏共享同一个 ECS World + 同一个渲染管线 → "你改的东西就是你玩的东西"。

### 4.2 ECS 是骨架

主要 schedule：
```
Startup → asset preload, scene load
PreUpdate → input gather, time tick
Update → game systems (player, sonar, BT, dialogue tick, ...)
PostUpdate → physics step, transform propagation
Render → camera collect, draw frame
```

核心组件类型（建议）：
- `Transform { translation, rotation, scale }`
- `Player`, `Camera`, `SonarEmitter`
- `BtEntity { state, awareness, kill_progress, ... }`
- `Marker { name }` —— 替代 GLB 里的 M_ 前缀
- `TriggerVolume { shape, on_enter, on_exit }`
- `Renderable { mesh, material }`
- `PointCloud { points: Vec<SonarPoint> }` —— 整个声呐 buffer 是一个组件
- `DialogueRunner { script_id, head, ... }`

System 顺序明确，无借用冲突，并行调度自动安排。

### 4.3 渲染管线：分层 forward+ + sonar 呈现层

两个并行的渲染层：

**Layer A · 真实世界 (PBR forward+)**
- Geometry pass：填 G-buffer subset (depth + normal)
- Light culling：cluster forward
- Shading：PBR (metallic-roughness)，IBL，directional light + shadow map
- Post：tone map (ACES) + simple bloom

**Layer B · 声呐呈现**
- GPU instanced billboard / point cloud
- 直接读 ECS 里 `PointCloud` 组件 → upload 到 GPU buffer
- 共用 Layer A 的 depth → 自动深度遮挡
- 颜色编码用 vertex color；雾化 / 衰减用 fragment uniform

切换：phase 1-5 主显 Layer B（暗底点云），phase 6 主显 Layer A（亮色真实世界）。
**关键**：Layer A 一直在写 depth，让 Layer B 的点不穿墙。这一点当前架构已经验证可行。

### 4.4 关卡文件（RON）

每个关卡一个 `.scene.ron`：

```ron
Scene(
    name: "earth_phase_1",
    glb: "assets/levels/earth/scene.glb",  // 几何来源
    entities: [
        Entity(
            name: "spawn_marker",
            components: [
                Marker(name: "M_spawn"),
                Transform(translation: (82.26, 6.25, 69.6), yaw: 63.1),
            ]
        ),
        Entity(
            name: "leak_tree_trigger",
            components: [
                TriggerVolume(shape: Sphere(radius: 8.0)),
                Transform(translation: (14.29, 2.32, 83.33)),
                OnEnter(script: "phase3.tree_leak_event"),
            ]
        ),
        Entity(
            name: "bt_spawn_01",
            components: [
                Marker(name: "M_leak_phantom_01"),
                Transform(translation: (49.09, 7.35, 64.62)),
                BtSpawner(mesh: "assets/entities/muddy_man.glb"),
            ]
        ),
    ],
    timeline: [
        OnEnter(script: "phase1.intro"),
        OnEnergyBelow(0.30, script: "phase1.low_energy"),
    ],
)
```

GLB 仍是**几何来源**，但所有"哪儿放 marker、什么触发哪段剧本"的元数据写在 RON 里。编辑器编辑的是 RON，不动 GLB。

### 4.5 剧本 DSL

文件 `scripts/phase1.ron` 或 `scripts/phase1.script`（自定义文本格式）：

```
phase phase1 {
    on_enter {
        play_music "music_relief"  
        wait 0.8s
        subtitle "外部供电——断了。" hold 2.6s
        wait 0.3s
        subtitle "我只剩这件套装的内部电池。" hold 3.0s
        wait 0.4s
        popup "长按左键 · 发射 Sonar 扫描" hold 5.0s
    }

    on_first_scan_bt {
        popup_override "系统提示：检测到威胁！\n长按右键解离威胁！\n不要暴露自己！" hold 6.5s
        play_music "music_threat"
    }

    on_energy_below 0.70 once {
        subtitle_override "电量 70%。" hold 2.2s
    }

    on_energy_below 0.30 once {
        subtitle_override "电量 30%——再不省着用我就完了。" hold 2.8s
        warning "电量危急 · 寻找补给"
    }

    on_death {
        advance_phase
    }
}
```

解析器解析一次成 AST，运行时按事件触发。**文件保存即热重载**：notify crate 监 fs event → 重新解析 → 替换该 phase 的脚本机。

进阶能力（按需）：
- 变量（local state per script run）
- 条件分支（看 phase / energy / 已死亡次数）
- 调用子段（go_to phase3.tree_leak）

### 4.6 资产管线 + 热重载

```rust
struct AssetServer {
    glbs: HashMap<PathBuf, Handle<Mesh>>,
    textures: HashMap<PathBuf, Handle<Texture>>,
    sounds: HashMap<PathBuf, Handle<Sound>>,
    scripts: HashMap<PathBuf, Handle<Script>>,
    fs_watcher: RecommendedWatcher,
}
```

- 资产用相对路径作 ID
- 加载是异步的（avoid block frame）
- watcher 监 `assets/` → 触发 reload
- ECS Renderable 持 `Handle<Mesh>` 而不是真实数据，reload 透明

### 4.7 编辑器（MVP 范围）

最小可用：

1. **Scene 视图**：第三人称漫游 + WASD + 鼠标右键拖看
2. **Outliner 面板**（egui）：当前场景所有 Entity 列表，点选高亮
3. **Inspector 面板**：选中 entity 后改 Transform / Marker name / 加组件
4. **Asset 面板**：列出 assets/ 下所有 GLB/wav/png/script，双击放进场景
5. **Script 编辑面板**：选 phase，编辑剧本，**Run preview** 按钮立刻播放
6. **菜单**：File / Save Scene、File / Reload All Assets、Play (F5 切到玩法模式)、Editor (F11 切回)

**不做**：动画曲线、节点蓝图、material editor、地形雕刻、Bevy/Unity 等价物。

## 6. 这一次必须丢的东西

### 6.1 整体架构
- `GameApp` 这个上帝 struct —— 拆成 N 个 ECS resource + 单一 System
- `match self.mode { Ship => ..., Earth => ... }` 这种模式分支 —— 改成 ECS state
- 散在 GameApp 里的所有 `Option<...>` 字段（dev_mode, leak_tree_state, ...） —— 改成组件
- `compute_spawn` 那种 trait-like 工具方法 —— 改成 Spawner system

### 6.2 渲染拼凑
- 三套并存的渲染上下文（miniquad / macroquad / ship::render）—— 统一到一个 wgpu 渲染图
- depth pre-pass 手动管理 —— 让渲染图自动调度
- macroquad 整个砍掉

### 6.3 内容管线 hack
- `_temp_Blender` 下几十个一次性 Python 脚本 —— 编辑器+ live preview 取代
- "改了 marker 要重导整个 GLB" —— RON 文件读 + GLB 只读几何
- `M_spawn_main` 在 (0,0,0) 的奇怪默认值 —— 没了 M_ 前缀就没了这个问题
- `tools/` 下七十几个 fix/inspect/process 脚本

### 6.4 字幕系统
- `narrative/mod.rs` 里 `match section { 5 => ..., 4 => ... }` 的硬编码 —— 全搬剧本文件
- `schedule` / `schedule_card` / `schedule_logo` / `schedule_popup` / `schedule_override` / `popup_override` —— 统一成 `cue!` 宏 + DSL 关键字
- 字幕 fade 等时间常量 —— 进剧本

### 6.5 平台 hack
- `GetAsyncKeyState` win32 调用 —— 用 winit input events
- `SetCursorPos` 钉光标 —— 用 winit `set_cursor_grab` 跨平台版
- `mem::forget(stream)` —— 用 kira 不需要这种 hack

### 6.6 调试遗物
- 左上角坐标 HUD（移到编辑器面板）
- DEV mode（T 键飞行）—— 改成编辑器内置漫游
- BT 测试 spawn 硬编码 —— 进 scene.ron
- 那些 `println!` —— 改 `tracing::info!`

## 7. 里程碑（建议 6 期 × 1-2 周/期 = 8-12 周）

### M4：剧本系统 (1.5 周)

### M5：玩法迁移 (2 周)

### M6：抛光 + 跨平台 (1-2 周)

## 8. 风险与备选

### 8.1 自写渲染 vs 用现成
- **风险**：6 周内自写 PBR + shadow 是激进的
- **备选**：用 `rend3` 或 `kajiya` 作为渲染层，专注 game logic
- **判断**：M2 第一周如果 PBR 没跑通，立即切 `rend3`，不死磕

### 8.2 编辑器无底洞
- **风险**：一旦做编辑器，需求会不停涨
- **备选**：M3 严格只做 Outliner/Inspector/AssetBrowser/SceneSave 四件，其它一律 vNext
- **判断**：每周 review 编辑器新需求列表，砍掉 8 成

### 8.3 剧本 DSL 过度设计
- **风险**：想做条件分支、变量、子段调用，最后变成半个 lua
- **备选**：M4 第一周只解析"线性命令序列 + on_X 事件"两种结构，禁止变量
- **判断**：变量真的需要时再加

### 8.4 ECS 学习成本
- **风险**：query 写法、system param、states 用法都不直觉
- **备选**：第一周写 5 个最简单的 system 找手感，复杂的留到 M5
- **判断**：如果 M1 末跑不通基本 ECS demo，回退 hecs（API 更直接）

## 9. 不变的核心

不管底座怎么换，下一个版本必须保留这四点。任何技术选择如果妨碍它们，就是错的选择。

1. **"你看见的，不是你看见的。"** —— 系统视角 vs 真实世界
2. **"回望发现自己一直被骗。"** —— 5→1 倒数才是真正的觉察
3. **声呐就是工具的视角** —— 玩家越用，越被绑死在系统的眼里
4. **"关掉它，才能看见真相。"** —— 反叛 = 不再依赖 Sonar = phase 6

这四点是 SonarJam 的灵魂。底座是为它服务的，不是反过来。

## 10. 给"未来 PA + 未来 Claude"的 cold-start 指南

如果你打开这份文档时是几个月以后、上下文全无：

1. 把 `desktop/sonar`（第一次重建）跑一遍，**那是手感原型**，确认 sonar 视觉配方好不好玩
2. 把 `SonarJam`（第二次重建）跑一遍，**那是叙事弧原型**，确认 5→0 的剧情节奏对不对
3. 然后看本文档 §1-9，知道第三次要解决什么、目标在哪
4. 从 M0 开始建新仓库（不是覆盖 SonarJam，是新仓库 `sonar-engine` 或类似名）
5. SonarJam 仓库保留作为内容档案库，别动

**别**把第三次叠在 SonarJam 上做。底座要换，从 0 开始更快。
