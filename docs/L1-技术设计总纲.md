## 1. 文档目的

本文档是当前项目的 L1 技术设计总纲。

本文档负责回答以下问题：

- 当前 MVP 要做成什么样的技术系统组合
- 当前技术路线如何服务已确认体验目标
- 哪些能力属于本阶段必须实现
- 哪些能力属于明确不做或后置
- 模块如何拆分
- 内容如何从 `Blender` 进入运行时
- 后续 L2 文档应如何继续细化

本文档不负责：

- 固化完整剧情文本
- 固化最终世界观解释
- 固化最终关卡内容细节
- 固化所有系统参数

## 2. 当前版本设计目标

### 2.1 总目标

交付一个可在 `Windows PC` 上运行的、基于 `Rust + macroquad` 的、具备完整主体验闭环的短篇可玩构建。

该构建必须至少实现以下体验闭环：

1. 第一人称进入极暗环境
2. 依靠声呐显影理解空间
3. 消耗有限资源推进
4. 资源耗尽或死亡后发生回溯
5. 回溯后保留可感知残留
6. 玩家意识到工具输出并非客观现实
7. 玩家遭遇“过去的自己”式威胁
8. 玩家进入后半段 `Echo Mode`
9. 玩家通过干预残留痕迹影响过去的自己
10. 玩家抵达至少一个可结束的最终状态

### 2.2 技术表达目标

当前版本的技术设计必须优先保证以下能力：

- 低成本但稳定的第一人称 3D 体验
- 可控的 `Blender -> 运行时` 内容路径
- 可被脚本化与数据化驱动的事件系统
- 支持“工具有偏见”这一主题的感知层
- 支持 `54321 -> 12345` 双段结构的状态机

## 3. 当前产品假设

本节不是创意定稿，而是当前技术设计所依赖的最低产品假设。若后续产品文档推翻这些假设，必须回改 L1。

### 3.1 已固定的体验假设

- 游戏为第一人称
- 世界默认处于极暗条件
- 玩家主要通过声呐显影认知空间
- 工具输出不是客观现实，而是经过系统翻译后的现实
- 死亡或资源耗尽不是单纯失败，而是触发回溯
- 回溯后会保留一部分残留痕迹
- 后半段存在与“过去的自己”相关的可交互阶段

### 3.2 当前技术所依赖但允许后续细化的假设

- 世界观层面的“谎言来源”尚可细化，但技术上必须支持“工具可能漏报、谎报、错误分类或偏移”
- 主空间应支持反复重读和逐步重解释
- 后半段应复用前半段大部分运行时资产与地图，而不是建设完全独立的第二套关卡技术路径

### 3.3 当前临时产品假设

以下假设用于降低 MVP 技术风险，后续可以在产品批准后调整：

- MVP 默认采用一个主场景而不是多关卡串联
- `12345` 后半段默认采用同场景 `Echo Mode`
- “过去的自己”默认以回声残留、偏差提示、假性威胁、干预痕迹等方式呈现，不要求复杂 NPC AI

## 4. MVP 边界

### 4.1 必须实现

当前 MVP 必须实现以下能力：

- 第一人称移动与观察
- 极暗场景下的声呐显影
- 有限能量池，默认不自动恢复
- 死亡/能量耗尽后的回溯
- 可见的残留记忆层
- 至少一种“工具撒谎”形式
- 至少一种“过去的自己”显现形式
- `Physical Mode -> Echo Mode` 的模式切换
- 至少一个有效结局

### 4.2 明确不在 MVP 内

以下能力不应进入当前 MVP，除非用户显式升级范围：

- 开放世界或大规模多区域探索
- 复杂敌人 AI 树
- 完整背包/装备系统
- 网络、多人、云存档
- 通用编辑器
- 程序化内容生成
- 重量级材质系统
- 完整本地化框架
- 复杂物理模拟

### 4.3 可后置但预留接口意识

以下能力可以后置，但 L2 设计应避免把未来扩展路径完全堵死：

- 多结局扩展
- 多场景切换
- 更多声呐模式
- 更丰富的音频驱动演出
- 更多残留类型

## 5. 总体技术路线

### 5.1 运行时路线

运行时采用 `macroquad` 作为唯一框架。

当前技术目标不是构建一套通用 3D 引擎，而是在 `macroquad` 内实现一个足够可控的轻量 3D 玩法框架，覆盖：

- 第一人称相机
- 主场景渲染
- 声呐点云/显影渲染
- 简化碰撞与交互
- 事件驱动切换

### 5.2 内容路线

正式空间内容来源于 `Blender`。

当前推荐路线为：

1. `Blender` 作为场景权威源
2. 导出 `GLB`
3. 运行时读取静态几何、标记点、命名规则和必要元数据
4. 以侧车数据文件补充玩法级配置

当前 L1 推荐的内容契约方向如下：

- 几何网格：来自 `GLB`
- 碰撞网格：来自 `GLB` 指定集合或命名规则
- 标记点：`Empty` / 节点命名
- 玩法配置：侧车 `RON` 或 `JSON`

最终格式由 L2《内容管线设计》定稿。

### 5.3 双模式路线

MVP 采用双模式路线：

- `Physical Mode`
- `Echo Mode`

`Physical Mode` 负责前半段体验：

- 真实移动
- 消耗资源
- 发射声呐
- 感知偏置世界
- 触发死亡/回溯

`Echo Mode` 负责后半段体验：

- 复用同一场景
- 不再完全遵守前半段交互规则
- 允许对残留痕迹进行删改、标记、扰动或诱导
- 以低成本方式表达“过去的自己在保护自己”

### 5.4 回溯路线

回溯不采用“完整物理历史回放”方案。

当前推荐路线为：

- 使用作者定义或系统定义的回溯锚点
- 死亡/耗尽后重置可重建状态
- 保留被授权持久化的残留层数据

也就是说，当前 MVP 的回溯本质上是：

- **运行时状态回滚**
- **局部持久化残留保留**

而不是全世界逐帧时间倒放。

## 6. 核心系统视图

### 6.1 系统清单

当前版本预期至少包含以下顶级系统：

- `app`：程序入口、全局生命周期、配置加载
- `game`：主状态机与场景切换
- `player`：第一人称控制器、输入、基础交互
- `world`：世界实体、标记点、碰撞、场景查询
- `content`：地图与侧车数据加载、解析与实例化
- `render`：基础 3D 渲染、UI 渲染、声呐渲染
- `sonar`：声呐发射、命中、显影、偏置规则
- `memory`：残留层、回溯保留、回声记录
- `narrative`：事件驱动、阶段切换、结局判定
- `audio`：基础音频、空间音、状态提示
- `ui`：能量、模式、最小引导信息
- `build`：运行配置与打包辅助

### 6.2 系统职责

#### `app`

负责：

- 启动
- 主循环绑定
- 配置装配
- 崩溃前最小日志输出

不负责：

- 具体玩法逻辑

#### `game`

负责：

- Game-wide 状态机
- 主模式切换
- 场景生命周期
- 重置与回溯编排

#### `player`

负责：

- 第一人称相机与移动
- 基础交互输入
- 模式差异下的可用操作

#### `world`

负责：

- 世界空间查询
- 碰撞信息
- 标记点与触发区
- 与内容实例的运行时表示

#### `content`

负责：

- 读取 `GLB`
- 读取侧车配置
- 将内容转为运行时世界结构

#### `render`

负责：

- 几何可视化
- 黑暗基底
- 声呐点云或等价显影层
- UI/HUD 合成

#### `sonar`

负责：

- 发射规则
- 命中与采样
- 能量消耗
- 可见结果生成
- 谎报/漏报/偏移等偏置逻辑入口

#### `memory`

负责：

- 回溯后可保留的数据
- 历次痕迹的残留表示
- Echo 层读取与写入

#### `narrative`

负责：

- 阶段推进
- 关键触发
- 模式切换
- 结局判定

#### `audio`

负责：

- 关键反馈音
- 声呐音
- 空间提示音
- 状态切换提示

#### `ui`

负责：

- 能量显示
- 模式状态
- 最小上下文提示

### 6.3 系统边界原则

系统边界应遵守以下原则：

- `narrative` 不直接承载渲染细节
- `render` 不直接决定剧情状态
- `content` 不直接拥有运行时主循环逻辑
- `memory` 不直接驱动物理碰撞
- `sonar` 输出“感知结果”，而非直接篡改世界基础几何

最后一条非常关键。当前技术总纲默认：

**世界基础空间与感知结果分层。**

这使“工具撒谎”可以通过感知层实现，而不强依赖真实几何被物理篡改。

## 7. 运行时状态机

### 7.1 顶级状态

当前推荐顶级状态如下：

- `Boot`
- `MainMenu` 或最小启动态
- `PhysicalMode`
- `DeathReset`
- `EchoMode`
- `Ending`

### 7.2 最小状态转移

推荐的 MVP 最小状态转移如下：

`Boot -> MainMenu -> PhysicalMode -> DeathReset -> PhysicalMode`

在满足叙事条件后：

`PhysicalMode -> EchoMode -> Ending`

### 7.3 状态设计要求

L2《运行时状态机设计》必须明确：

- 哪些数据在状态间保留
- 哪些数据在回溯时重置
- 哪些数据在 `EchoMode` 中只读
- 哪些数据可以被 `EchoMode` 改写

## 8. 感知模型与“有罪的工具”

### 8.1 技术设计目标

当前技术上不要求先定义唯一的叙事真相，但必须支持：

- 工具输出与基础世界之间存在系统性偏差
- 偏差不是纯随机，而是可设计、可脚本化、可重演

### 8.2 当前支持的偏差类型

MVP 至少应支持以下一种或多种偏差类型：

- 漏报：某些对象或区域不被声呐正确显影
- 谎报：显影出不存在或被夸大的威胁轮廓
- 偏移：显影位置与真实位置存在稳定偏差
- 错误分类：同一对象被渲染为错误语义层

### 8.3 设计原则

- 偏差必须服务体验，而不是纯粹随机折磨
- 偏差必须具备作者控制入口
- 偏差必须可被叙事或模式切换重新解释
- 偏差应优先作用于感知结果层，而不是优先破坏基础世界层

## 9. 残留层与回声层

### 9.1 残留层

残留层是前半段回溯后仍保留的那部分信息。

MVP 中残留层至少应支持：

- 历史显影痕迹
- 关键提示残影
- 过去自己的存在感

### 9.2 回声层

回声层是 `Echo Mode` 操作对象。

MVP 中回声层不要求完整模拟过去的整段行为，但必须支持：

- 读取被保留的痕迹
- 局部干预某些痕迹
- 影响后续一条作者定义路径的解释结果

### 9.3 MVP 推荐实现策略

当前推荐采用“作者控制的半数据驱动残留系统”，而非“全历史录像系统”。

原因：

- 成本更低
- 行为更可控
- 更适合 `48 小时` 交付

### 10.1 空间资产

- 运行时导入静态网格

### 10.2 标记点与集合约定

- 结局标记点

- `RON`
- `JSON`

## 14. 主要风险

### 14.1 范围风险

`12345` 后半段若按“完整第二游戏”实现，会超过 MVP 承载能力。

当前缓解策略：

- 复用主场景
- 复用大部分系统
- 将后半段收敛为 `Echo Mode`

### 14.2 内容管线风险

`macroquad` 下 3D 内容导入与运行时结构若不提前收口，容易在后续返工。

当前缓解策略：

- 尽快形成 L2《内容管线设计》
- 尽快冻结最小导入契约

### 14.3 创意演进风险

世界观与后半段表达仍在细化，可能影响偏差机制与事件系统。

当前缓解策略：

- L1 只锁技术能力，不锁剧情细节
- 在 L2 中把叙事接口与剧情文本分离

### 14.4 时间风险

`48 小时` 约束下，任何“更通用但更慢”的方案都可能破坏交付。

当前缓解策略：

- 严格按阶段推进
