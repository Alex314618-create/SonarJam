## 1. 文档目的

本文档定义当前 MVP 阶段的正式内容管线。

其目标是回答以下问题：

- 作者在 `Blender` 中应该如何组织地图
- 哪些内容由 `GLB` 承载，哪些内容由侧车配置承载
- 运行时如何将内容导入为可消费的关卡结构
- 如何支持“真实空间”和“系统解释后的空间”共存
- 当前阶段明确支持什么，不支持什么

本文档不负责：

- 定义剧情文本内容
- 定义最终叙事碎片文案
- 定义所有触发逻辑细节
- 设计复杂编辑器

## 2. 设计目标

### 2.1 管线目标

当前内容管线必须满足以下目标：

- 对作者足够简单
- 对运行时足够稳定
- 对 `48 小时 GameJam` 交付足够可控
- 支持单一主场景被反复重读
- 支持“感知解释层”与“基础空间层”分离

### 2.2 技术目标

当前内容管线必须支持：

- 静态空间几何导入
- 独立碰撞几何导入
- 点型与区域型标记点导入
- 侧车规则对空间对象的绑定
- `PhysicalMode` 与 `EchoMode` 对同一空间对象的差异解释

### 2.3 范围控制目标

当前内容管线必须刻意保持狭窄。

本设计不是通用 3D 关卡导入器。它只服务当前项目 MVP。

## 3. 总体方案

### 3.1 正式管线方案

当前正式内容管线采用：

**`Blender -> GLB + JSON 侧车 -> 运行时 StaticLevel`**

其中：

- `Blender` 是空间内容的权威源
- `GLB` 负责承载空间几何与标记节点
- `JSON` 侧车负责承载玩法规则、感知偏差规则与事件绑定
- 运行时将导入结果扁平化为统一的 `StaticLevel`

这条分工必须保持严格。`JSON` 不得重复声明空间坐标、几何拓扑或对象放置结果；`GLB` 不得承担复杂玩法规则编排。

### 3.2 分层原则

内容分为以下三层：

#### 空间层

定义：

- 真实空间位置
- 静态几何
- 碰撞几何
- 标记点与区域位置

权威来源：

- `GLB`

#### 关卡规则层

定义：

- 标记点语义
- 模式切换规则
- 偏差规则
- 触发器与事件绑定
- Echo 干预点

权威来源：

- `level.json`

#### 叙事引用层

定义：

- 文本 ID
- 日志 ID
- 回声碎片 ID
- 系统告警 ID

权威来源：

- 当前不直接定义在本管线文档中
- 后续由 L2-06 叙事事件与触发设计负责

### 3.3 总体原则

当前内容管线遵守以下原则：

- `Blender` 决定“东西在哪”
- 侧车配置决定“东西意味着什么”
- 运行时消费的是“关卡数据”，不是“DCC 场景”

## 4. 关卡包结构

### 4.1 推荐结构

当前推荐关卡包结构如下：

```text
content/
  levels/
    earth_return_01/
      scene.glb
      level.json
```

### 4.2 结构说明

- `scene.glb`：正式空间内容
- `level.json`：正式规则配置

### 4.3 目录式方案的原因

采用目录式关卡包，而不是将 `GLB` 与规则文件分散存放，原因如下：

- 便于协作者快速定位同一关卡的全部资产
- 便于未来扩展额外文件
- 更利于 agent 在无 context 场景下理解内容边界
- 降低多关卡时的引用混乱风险

## 5. Blender 作者约定

### 5.1 基本原则

Blender 仅负责当前项目需要的最小集合内容。

作者不得假设运行时会理解：

- 任意材质网络
- 任意动画
- 任意约束系统
- 任意自定义节点
- 任意 collection 嵌套逻辑

### 5.2 支持的对象类别

当前 MVP 中，运行时只正式接受以下类别：

- 静态渲染网格
- 静态碰撞网格
- 点型标记点
- 区域型辅助对象

### 5.3 不支持的内容

当前 MVP 中，不正式支持以下内容：

- 骨骼动画
- 复杂灯光还原
- 粒子系统导入
- 曲线驱动玩法
- 布尔历史依赖
- 动态刚体信息导入
- 通用骨架角色

### 5.4 集合使用原则

当前 authoring 至少应存在以下顶级 collections：

- `Render`
- `Collision`
- `Markers`
- `Volumes`

`Blender Collections` 可以作为作者组织手段，但不应作为运行时唯一契约来源。

换言之：

- Collection 用于作者管理
- Object Name 用于运行时权威识别

## 6. 命名约定

### 6.1 命名原则

对象命名必须：

- 稳定
- 全局唯一
- 可被程序解析
- 不依赖中文描述

当前推荐采用 ASCII 命名。

### 6.2 对象前缀

当前推荐采用以下前缀：

- `R_`：Render Mesh，渲染网格
- `C_`：Collision Mesh，碰撞网格
- `M_`：Marker，点型标记
- `V_`：Volume，区域型标记

### 6.3 Marker 命名建议

当前推荐至少支持以下语义族：

- `M_spawn_*`
- `M_reset_*`
- `M_trigger_*`
- `M_echo_*`
- `M_truth_*`
- `M_end_*`

### 6.4 Volume 命名建议

当前推荐至少支持以下语义族：

- `V_bias_*`
- `V_truth_*`
- `V_block_*`
- `V_trigger_*`

### 6.5 权威 ID

当前 MVP 中，`Blender object name` 不应被视为 canonical gameplay ID。

当前约定如下：

- `Blender object name` 是作者可见定位符
- gameplay-relevant object 必须拥有稳定 authoring ID
- 侧车配置应引用稳定 authoring ID，而不是直接把 object name 当作最终语义标识

当前推荐的稳定 authoring ID 字段名为：

- `sj_id`

当前推荐的对象种类字段名为：

- `sj_kind`

实现路径可以有两种：

1. 运行时直接读取 `GLB extras`
2. 导出期生成 `scene index` 供运行时消费

本 L2 固定的是契约，不固定具体技术实现路径。具体采用哪条路径由 L3 实施计划决定。

## 7. GLB 导出契约

### 7.1 导出目标

`GLB` 导出必须至少保留：

- 静态网格
- 节点名称
- 节点变换
- gameplay-relevant object 的稳定 authoring ID
- gameplay-relevant object 的对象种类

### 7.2 当前依赖信息

运行时当前依赖以下导出信息：

- object name
- transform
- mesh geometry
- gameplay-relevant object 的稳定 authoring ID
- gameplay-relevant object 的对象种类

### 7.3 当前不依赖的信息

运行时当前不依赖：

- Blender 灯光
- Blender 相机
- 复杂材质参数
- 动画轨道
- 约束器行为

### 7.4 导出一致性要求

内容作者应保证：

- 坐标系使用统一约定
- 不存在未应用或不可预测的异常缩放
- 同一对象的用途与命名前缀一致

最终坐标约定由实现验证后在本文件升级版本中补充精确条款。

### 7.5 导出前约束

对于 gameplay-relevant object，作者应保证：

- 已应用 transform
- 不使用负缩放
- 不依赖运行时求值的复杂约束
- 仅使用静态、可烘焙结果

若某对象不能满足这些约束，则不得作为 gameplay-relevant object 进入正式内容路径。

## 8. 侧车配置职责

### 8.1 侧车文件格式

当前 MVP 默认采用：

- `level.json`

采用 `JSON` 的原因：

- 可读性高
- 调试友好
- 对人工与 agent 都足够直接
- 当前阶段不值得为表达性引入更高认知成本

### 8.2 侧车文件职责

侧车文件负责承载：

- 关卡级 ID
- `scene.glb` 路径绑定
- stable ID 到对象语义的绑定
- 模式切换规则
- 感知偏差规则
- Echo 干预规则
- 触发器到事件的绑定

### 8.3 侧车文件不负责

侧车文件不负责重复描述：

- 顶点数据
- 网格坐标
- marker 坐标
- volume 几何
- 空间对象的真实放置结果

### 8.4 当前建议字段

当前建议 `level.json` 至少具备如下顶层字段：

- `level_id`
- `scene`
- `semantic_objects`
- `spawn_object`
- `reset_objects`
- `ending_objects`
- `mode_switches`
- `bias_rules`
- `echo_bindings`
- `trigger_bindings`

最终 schema 由实现验证后在后续版本中精化。

## 9. 感知解释层契约

### 9.1 设计动机

当前创意要求：

- 真实空间存在
- 系统会对真实空间做有偏见的翻译
- 同一空间在 `PhysicalMode` 与 `EchoMode` 下可被不同解释

因此，内容管线必须把“空间”和“解释”分开。

### 9.2 感知解释层的职责

感知解释层负责定义：

- 哪些对象会被漏报
- 哪些对象会被谎报
- 哪些对象会被偏移
- 哪些对象会在不同模式下被重新标记

### 9.3 解释层的绑定方式

当前推荐：

- `GLB` 提供空间对象与稳定 authoring ID
- `level.json` 通过稳定 ID 绑定解释规则

示例：

- `V_bias_hall_a` 绑定为某一章中的高偏差区域
- `M_echo_hint_01` 在 `PhysicalMode` 下被系统标记为敌意，在 `EchoMode` 下被重新解释为友方痕迹

### 9.4 解释层的最小规则能力

MVP 中至少应支持以下一种或多种：

- `omit`
- `offset`
- `threat_relabel`
- `echo_reveal`

这些是运行时行为能力，不等于必须在第一版全部做满。

## 10. 运行时导入模型

### 10.1 目标运行时结构

运行时不应保留复杂 DCC 场景图。

导入后应尽快扁平化为统一关卡结构，例如：

- `StaticScene`
- `CollisionWorld`
- `AnchorTable`
- `SemanticObjects`
- `PerceptionProfiles`
- `SessionState`

### 10.2 StaticLevel 原则

运行时消费的是 `StaticLevel`，而不是 Blender 原始结构。

该原则的好处：

- 简化运行时查询
- 简化碰撞与触发器
- 简化偏差规则绑定
- 降低内容导入与玩法逻辑耦合度

### 10.3 Marker 表示

Marker 在运行时应至少包含：

- `authoring_id`
- `position`
- `rotation`（如有）
- `kind`
- `node_name`

其中：

- `authoring_id` 来自稳定 authoring ID
- `kind` 可由命名前缀和侧车语义共同决定

### 10.4 Volume 表示

Volume 在运行时应至少包含：

- `authoring_id`
- `shape`
- `transform`
- `kind`
- `node_name`

当前 MVP 推荐只支持简单体积表示，不支持复杂布尔区域组合。

## 11. 失败模式与验证

### 11.1 必须定义的失败模式

L3 实现与运行时必须能处理以下失败模式：

- `scene.glb` 不存在
- `level.json` 不存在
- `GLB` 中缺少关键 gameplay object
- 侧车引用了不存在的 stable ID
- stable ID 重复
- 前缀非法
- 碰撞网格缺失
- 模式绑定引用了不存在的 volume 或 marker
- gameplay object 缺少稳定 authoring ID
- gameplay object 缺少对象种类

### 11.2 错误处理原则

错误处理应遵守：

- 尽早失败
- 错误信息可定位
- 不静默吞错
- 不在错误输入上继续构建不一致场景

### 11.3 Fatal 与 Warning 规则

以下情况应视为 `Fatal`：

- duplicate stable ID
- 缺少 stable ID
- 缺少 `spawn` / `reset` 等关键对象
- 侧车引用未知 stable ID
- gameplay-relevant object 使用不受支持的导出特性
- collision 语义对象缺少实际碰撞代理

以下情况可以视为 `Warning`：

- 未使用的语义对象绑定
- 未被侧车使用的非关键 marker
- render-only mesh 使用不被运行时消费的复杂材质信息
- 作者组织层面的无害 collection 冗余

### 11.4 双阶段验证

当前内容管线应采用双阶段验证：

1. 导出期验证
2. 加载期验证

导出期验证优先发现：

- authoring ID 缺失
- authoring ID 重复
- 非法对象类别
- 基础 authoring 结构错误

加载期验证优先发现：

- 文件缺失
- JSON 绑定失效
- 运行时 schema 不匹配
- 最终关卡结构无法构建

### 11.5 最小验证清单

内容管线实现后，至少应验证：

1. 可以加载最小示例关卡
2. 可以解析一个渲染网格
3. 可以解析一个碰撞网格
4. 可以解析一个带稳定 authoring ID 的 `spawn marker`
5. 可以解析一个 `bias volume`
6. 可以从 `level.json` 绑定一条偏差规则
7. 当 stable ID 缺失、重复或绑定失效时会输出明确错误

## 12. 当前明确不支持项

以下内容明确不在当前 L2-02 支持范围内：

- 第二套独立 authored `EchoMode` 地图
- 多层嵌套 prefab 管线
- 自动从材质推导玩法语义
- 从渲染网格自动生成全部碰撞
- 通用剧情图编辑器
- 复杂路径网络
- 动态导航网格导入
- 多场景流式加载
- 程序化几何内容
- 仅通过名字编码全部玩法逻辑

## 14. 决策记录

### D-001

- 决策：采用 `GLB + JSON` 侧车的混合方案
- 可选方案：纯 `GLB` 元数据、纯文本关卡描述、其他配置格式
- 未选原因：纯 `GLB` 不利于复杂规则表达；纯文本不适合空间 authoring；其他配置格式在 MVP 阶段没有显著收益
- 失效条件：若 `JSON` 无法承载实际规则需求，且团队明确批准升级格式

### D-002

- 决策：使用目录式关卡包
- 可选方案：分散式资源布局
- 未选原因：目录式更利于追踪、协作和 agent 理解
- 失效条件：若后续发现资源体量和复用模式要求集中式资产目录

### D-003

- 决策：Blender object name 仅作为作者定位符，gameplay canonical ID 采用稳定 authoring ID
- 可选方案：直接使用 object name 作为 canonical ID
- 未选原因：重命名、复制与 Blender 自动后缀会让 name-based 方案脆弱且难审计
- 失效条件：若稳定 authoring ID 的落地成本在 MVP 中被证明不可接受，且经批准切换到更窄方案

### D-004

- 决策：空间层与感知解释层分离
- 可选方案：直接修改基础几何表达所有欺骗效果
- 未选原因：几何层与感知层混杂会增加运行时耦合并削弱“工具有罪”的主题表达
- 失效条件：若实际演出需求证明某些关键效果必须直接修改基础世界层

### D-005

- 决策：`GLB` 承载 spatial truth，`JSON` 承载 semantic/perception truth
- 可选方案：将语义直接编码进纯命名规则或将空间细节重复写入 `JSON`
- 未选原因：纯命名规则不可审计且脆弱；在 `JSON` 中重复空间事实会导致 split-brain 漂移
- 失效条件：若运行时或工具链能力迫使关卡清单由导出期自动生成额外索引文件

## 15. 待决事项
